I. Descripción del Sistema y Alcance

El Sistema de Información para la Gestión de Reservas del Hostal Apu Qhawarina tiene como objetivo automatizar, centralizar y optimizar el proceso de reservas, registro y control de huéspedes, asegurando una administración eficiente de habitaciones y servicios del hostal.

Alcance del sistema

  • Permitir que los huéspedes realicen reservas en línea o de forma presencial.
  • Facilitar la validación automática de disponibilidad de habitaciones.
  • Registrar datos personales y mantener historial de huéspedes.
  • Controlar el estado y ocupación de las habitaciones en tiempo real.

1.2. Funcionalidades del Sistema

A. Funcionalidades Clave (Obligatorias)

  • Gestión de Reservas: Registrar, modificar, validar o cancelar reservas.
  • Validación y Confirmación: Verifica disponibilidad y genera confirmaciones.
  • Gestión de Habitaciones: Control del estado (disponible, ocupada, mantenimiento).
  • Pre Check-In y Check-In: Registro previo y validación a la llegada.
  • Gestión de Usuarios y Roles: Accesos según tipo de usuario.
  • Generación de Reportes: Emite reportes de ocupación e ingresos.

B. Funcionalidades Secundarias (Opcionales / Futuras)

  • Reservas en línea mediante portal web o aplicación móvil.
  • Notificaciones automáticas vía correo o WhatsApp.
  • Módulo de estadísticas y reportes avanzados.

1.3. Identificación de Usuarios Principales (Actores)

Actor / Usuario Rol dentro del Sistema
Huésped (Cliente) Realiza reservas, pagos y pre check-in.
Recepcionista Administra reservas, realiza check-in/check-out.
Administrador / Gerente Supervisa operaciones, gestiona usuarios y reportes.
Sistema de Reserva (Interno) Automatiza validaciones y notificaciones.

II. Requisitos No Funcionales

Seguridad

El sistema debe garantizar la protección de datos mediante autenticación, encriptación y respaldos automáticos.

Justificación: Evita accesos no autorizados y pérdida de información.

Rendimiento

El sistema debe procesar operaciones comunes en menos de 2 segundos.

Justificación: Mejora la eficiencia y experiencia del usuario.

Usabilidad

Debe contar con una interfaz intuitiva y fácil de usar.

Justificación: Minimiza errores y facilita el aprendizaje.

Compatibilidad

Accesible desde distintos navegadores y dispositivos.

Justificación: Favorece la flexibilidad y disponibilidad.

Mantenibilidad

Fácilmente actualizable y escalable.

Justificación: Facilita futuras mejoras sin afectar el sistema.

III. Diagramas del Sistema

Diagrama de Casos de Uso - Reserva

1. Descripción general del diagrama de reserva

El diagrama de casos de uso representa las interacciones entre los actores y el sistema de reserva del hostal, mostrando las funcionalidades principales desde la perspectiva del usuario. El objetivo principal del sistema es permitir que los huéspedes realicen reservas de hospedaje de manera sencilla, gestionando los datos personales, las fechas, las habitaciones.

2. Actores identificados

  • Huésped: Persona externa que desea realizar una reserva en el hostal. Interactúa directamente con el sistema para buscar, seleccionar y confirmar su hospedaje.
  • Sistema de Reserva: Representa la lógica interna o módulo automatizado que valida y gestiona la información de las reservas.
  • Recepcionista: Empleado del hostal encargado de recibir la notificación de nuevas reservas y realizar la gestión física del hospedaje.

3. Casos de uso principales

  • Buscar hospedaje: El huésped accede al sistema para iniciar una búsqueda de habitaciones disponibles según sus necesidades.
  • Ver opciones de hospedaje: El sistema muestra diferentes tipos de habitaciones, precios y servicios.
  • Seleccionar hospedaje: El huésped elige una opción específica de alojamiento según sus preferencias.
  • Ingreso a la plataforma: El huésped accede al sistema de reservas (portal web o aplicación).
  • Ingreso de datos personales: El huésped registra su información (nombre, documento, contacto, etc.) para completar la reserva.
  • Seleccionar habitación: El huésped elige la habitación deseada dentro de las disponibles.
  • Seleccionar fechas: El huésped indica las fechas de entrada y salida para la reserva.
  • Realizar reserva: El huésped confirma los datos ingresados y solicita la reserva formalmente. Este caso de uso incluye la selección de habitación y de fechas.
  • Pagar reserva: El huésped realiza el pago correspondiente.
  • Confirmación de reserva: El sistema genera una confirmación de la reserva (número, fechas, habitación, monto, etc.) y la envía al huésped.
  • Validación de la reserva: El sistema valida la información del huésped, la disponibilidad y el pago antes de confirmar la reserva.
  • Notificación de la reserva: El sistema notifica automáticamente al recepcionista sobre la nueva reserva registrada.

4. Relaciones entre casos de uso

  • Relación <<include>>:
    • “Realizar reserva” incluye “Seleccionar habitación” y “Seleccionar fechas”.
    • “Confirmación de reserva” incluye “Validación de reserva”.
  • Dependencias funcionales:
    • “Pagar reserva” ocurre después de “Realizar reserva”.
    • “Confirmación de reserva” se ejecuta tras el pago exitoso.
    • “Notificación de la reserva” se activa automáticamente tras la confirmación.

5. Flujo general del proceso

  1. El huésped busca y visualiza las opciones de hospedaje.
  2. Selecciona una habitación y las fechas deseadas.
  3. Ingresa su información personal.
  4. Realiza la reserva, la cual es validada por el sistema.
  5. Efectúa el pago correspondiente.
  6. El sistema genera la confirmación de la reserva.
  7. El recepcionista recibe la notificación para registrar la llegada del cliente.

6. Conclusión

Este diagrama de casos de uso permite visualizar de forma clara la interacción entre los usuarios y el sistema. El huésped es el actor principal que ejecuta la mayoría de las funciones, mientras que el Sistema de Reserva y el Recepcionista tienen un papel de validación y gestión administrativa. El diseño asegura un flujo lógico de reserva que abarca desde la búsqueda del hospedaje hasta la confirmación final, garantizando que la información fluya correctamente entre los diferentes actores.

Diagrama de Casos de Uso - Cliente

1. Descripción general del diagrama

Este proceso tiene como finalidad facilitar el ingreso del cliente al hostal antes de su llegada, permitiéndole registrar y enviar su información personal para agilizar la atención en recepción. El sistema, junto con el recepcionista, se encarga de validar la reserva, guardar los datos y actualizar el estado de las habitaciones y reservas en función de la llegada del huésped.

2. Actor principal

  • Huésped (Cliente): Es la persona que ha realizado una reserva y que interactúa con el sistema para completar su proceso de pre check-in y posterior check-in.

3. Casos de uso relacionados con el cliente

  • Realizar Pre Check-in: El huésped ingresa al enlace proporcionado por el hostal para adelantar el registro antes de su llegada. Aquí completa sus datos personales y los detalles de su reserva.
  • Enviar la información del Pre Check-in: Una vez completado el formulario, el huésped envía la información al sistema de reservas, que se encargará de almacenarla y validarla.
  • Realizar el Check-in: Al llegar al hostal, el huésped confirma su presencia. Este proceso puede implicar la firma digital o física del registro de ingreso.

4. Relaciones (<<include>>)

  • “Realizar Pre Check-in” <<include>> “Enviar la información del Pre Check-in”
    ➜ Significa que el envío de la información es una acción obligatoria dentro del proceso de pre check-in.
  • “Realizar el Check-in” <<include>> “Validar reserva” (a través del sistema o recepcionista)
    ➜ Antes de que el huésped pueda completar su check-in, el sistema valida que su reserva esté vigente.

5. Flujo del proceso desde la perspectiva del huésped

  1. El huésped recibe un enlace de pre check-in (enviado por el sistema o el recepcionista).
  2. Ingresa al enlace y realiza el pre check-in, completando su información.
  3. Envía sus datos personales (nombre, documento, contacto, etc.) al sistema.
  4. El sistema almacena y valida los datos con la reserva existente.
  5. Al llegar al hostal, el huésped realiza su check-in presencial, finalizando el proceso de ingreso.

6. Conclusión

Desde la perspectiva del cliente (huésped), el sistema ofrece una experiencia más rápida y eficiente, permitiéndole registrarse anticipadamente y reducir tiempos al momento de su llegada. El diagrama refleja cómo el huésped interactúa activamente con el sistema en los pasos de pre check-in y check-in, mientras que las tareas de validación y actualización son gestionadas automáticamente por el sistema o con apoyo del recepcionista.
